## 1. プロジェクト概要

### 1.1 プロジェクト名

宮崎マリンスポーツ情報集約PWA

### 1.2 目的

宮崎県内の主要マリンスポーツエリア（木崎浜・青島・日南海岸）を利用するサーファー・SUP・ボディボード等の利用者に対し、意思決定に必要な情報（波・風・潮汐・天候・混雑状況等）を**一元集約**し、**要約・スコアリングした形で提供**するPWA（Progressive Web App）を構築する。

### 1.3 対象エリア（初期スコープ）

- 木崎浜（宮崎市）
- 青島（宮崎市）
- 日南海岸（日南市）

【推奨】将来的な対象エリア拡張（延岡・都農等）は本仕様のデータ構造を拡張する形で対応可能な設計とする。

### 1.4 対象ユーザー

- マリンスポーツ利用者（一般ユーザー、無料/有料会員）
- 情報提供者（現地ショップ・インストラクター等、Phase4で制度化）
- 運営者（管理者ロール）

### 1.5 プロジェクトの非目標（Non-Goals）

以下は本プロジェクトの範囲外であることを明確にする。

- 【必須】独自の気象予測AI・モデルを開発しない。既存の無料気象・海洋データを統合・スコアリング・要約するのみ。
- 【必須】有料AI API（例：OpenAI API、Claude API等の従量課金API）を運営コストに組み込まない。
- 【必須】運営コストが発生する有料インフラ・有料SaaSを原則使用しない（Phase3のStripeを除く。Stripeは決済発生時のみ課金される従量制であり、待機コストは0円）。

### 1.6 コア原則

1. **運営コスト0円**：無料枠のAPI・無料ホスティング（Cloudflare Pages）のみで運用可能な設計とする。
2. **情報の正直な要約**：予測データは「当たるとは限らない参考情報」であることを常にUI上で明示する（詳細は06_SECURITY_AND_PRIVACYおよび法務ドラフト参照）。
3. **段階的拡張**：Phase1（基盤）→Phase5（法務）まで、後方互換性を保ちながら段階的に機能を追加する。
4. **単一正本管理**：本ドキュメントを唯一の正本とし、実装側（`miyazaki-marine-app`）とは完全分離する。

## 3. Phase 1：基盤仕様

### 3.1 Phase1の目的

PWAとしての基本的な情報集約・表示機能を提供する土台を構築する。Phase2以降（AIスコアリング、Stripe、情報提供者制度、法務）はすべてPhase1の基盤の上に積み重ねる。

### 3.2 技術スタック【必須】

| レイヤー | 技術 | 補足 |
|---|---|---|
| フロントエンド | React + Vite | SPAとして構築 |
| ホスティング | Cloudflare Pages | 無料枠での運用を前提。Cloudflare Pages Functionsを使う場合も無料枠内に収める |
| PWA対応 | Service Worker + Web App Manifest | オフラインキャッシュ、ホーム画面追加対応 |
| データ取得 | 無料公開API・無料データソースのみ | 詳細は04_DATA_SPEC.md参照 |
| 状態管理 | 実装Repository側で決定（本仕様書では技術選定を強制しない） | 例：React Context / Zustand等、軽量なもので良い |
| ビルド・デプロイ | GitHub連携によるCloudflare Pages自動デプロイ | 08_DEPLOYMENT.md参照 |

【必須】有料AI API（従量課金のLLM API等）をアプリケーションの実行時パスに含めない。

### 3.3 対応デバイス・ブラウザ【必須】

- スマートフォン（iOS Safari / Android Chrome）を第一優先とする
- タブレット・PCブラウザでも表示崩れがないレスポンシブ対応とする
- 【推奨】オフライン時は最後に取得したデータをキャッシュ表示し、「オフライン・最終更新時刻」を明示する

### 3.4 画面構成（Phase1範囲）【必須】

1. **トップ画面（スポット一覧）**
   - 木崎浜・青島・日南海岸のカード表示
   - 各カードに現在の簡易コンディション表示（Phase2でスコア表示に接続）
2. **スポット詳細画面**
   - 波・風・潮汐・天候の生データ表示（Phase1時点ではスコアリングなしの生データ表示でも可）
   - 時間帯別（例：3時間おき）の予測データ表示
3. **設定画面**
   - お気に入りスポット登録
   - 通知設定（Phase1ではUIのみでも可。実際の通知配信はPhase以降で検討）

【任意】上記以外の画面（マップ表示、複数スポット比較表示等）はPhase1では任意機能とする。

### 3.5 データ取得の基本方針【必須】

- 無料公開APIから取得したデータは、取得元・取得時刻を記録し、UI上で「最終更新：HH:MM」を必ず表示する。
- APIごとの利用規約（利用回数制限、クレジット表記義務等）を`04_DATA_SPEC.md`に明記し、遵守する。
- 複数データソースを使う場合、データソース間で単位・座標系・タイムゾーンの不整合が生じないよう正規化処理を行う。

### 3.6 非機能要件（Phase1範囲）

| 項目 | 要件 |
|---|---|
| パフォーマンス | 【推奨】初回表示（First Contentful Paint）3秒以内（4G相当の回線を想定） |
| 可用性 | 【推奨】無料枠の制約上、100%の可用性は保証しないが、障害時は「データ取得エラー」を明示しユーザーを混乱させない |
| セキュリティ | 【必須】APIキーをクライアントコードに直接埋め込まない。必要な場合はCloudflare Pages Functions等のサーバーサイド経由で秘匿する（詳細は06_SECURITY_AND_PRIVACY.md） |
| アクセシビリティ | 【推奨】色のみに依存した情報表現を避け、コンディションスコア等はアイコン・テキストも併記する |
| 国際化 | 【任意】Phase1では日本語のみで良い。多言語対応は将来検討 |

### 3.7 Phase1の完了条件（Definition of Done）

- 【必須】3スポット（木崎浜・青島・日南海岸）の一覧・詳細画面が表示できる
- 【必須】無料データソースから取得した波・風・潮汐・天候の生データが表示される
- 【必須】PWAとしてホーム画面に追加でき、オフライン時に最終キャッシュが表示される
- 【必須】Cloudflare Pagesへの自動デプロイが機能する
- 【推奨】お気に入りスポット機能が動作する

## 4. Phase 2：AIスコアリング仕様

### 4.1 Phase2の目的と原則【必須】

Phase2では、Phase1で取得した無料の生データ（波・風・潮汐・天候等）を統合し、ユーザーが一目で判断できる「コンディションスコア」と「要約テキスト」を生成する。

【必須】本プロジェクトでは以下を厳守する。

- **独自の気象予測モデル・機械学習モデルを新規開発しない。** あくまで既存の無料データソースが提供する予測値をそのまま入力として使う。
- **有料AI API（LLM API等の従量課金サービス）を用いた自動要約・自動生成を行わない。** 要約テキストは、あらかじめ定義したルールベース・テンプレートベースの文章生成ロジックで実装する。
- スコアリングロジックはすべて**ルールベース（if-then／重み付け加算等）**で実装し、判断根拠を人間が説明できる状態を保つ。
- スコアは「絶対的な保証」ではなく「参考情報」であることを、UI・法務ドラフト（Phase5）双方で明示する。

### 4.2 コンディションスコアの構成要素【必須】

コンディションスコアは、以下の要素を統合して算出する（各要素の具体的な取得元は`04_DATA_SPEC.md`に定義する）。

| 要素 | 内容 | Phase1での扱い |
|---|---|---|
| 波高・周期 | サーフィン・SUP適性に直結 | 生データとして表示済み |
| 風向・風速 | オンショア／オフショアの判定に使用 | 生データとして表示済み |
| 潮汐 | 満潮・干潮のタイミング、スポットごとの適正潮位 | 生データとして表示済み |
| 天候・降水 | 視界・安全性への影響 | 生データとして表示済み |
| 水温（取得可能な場合） | 装備判断の参考情報 | 【任意】取得元があれば追加 |

### 4.3 スコアリングロジックの設計方針【必須】

1. 各要素について、スポットごとに「適正レンジ」を`04_DATA_SPEC.md`で定義する（例：木崎浜のサーフィン適性は波高0.5m〜1.5mを標準適正レンジとする、等）。
2. 各要素の実測値・予測値が適正レンジに対してどの程度合致するかを0〜100のスコアとして算出する（レンジ内なら高スコア、レンジ外に離れるほど減点）。
3. 各要素スコアに重み（Weight）を設定し、加重平均で総合コンディションスコア（0〜100）を算出する。重みの初期値は`04_DATA_SPEC.md`にて仮設定し、運用開始後に`DECISIONS.md`で調整履歴を記録しながら見直す。
4. 総合スコアを5段階（例：最高・良好・普通・やや不向き・不向き）にラベリングし、UI上はラベル＋スコア数値の両方を表示する。

【推奨】スポーツ種別（サーフィン／SUP／ボディボード等）によって適正レンジ・重みが異なることを踏まえ、Phase2以降はスポーツ種別ごとにスコアを出し分けられる設計とする。Phase1のデータ構造がこれを阻害しないよう、`04_DATA_SPEC.md`側でスポーツ種別のパラメータ化を行う。

### 4.4 要約テキスト生成の設計方針【必須】

- 要約テキストは、スコアリング結果と各要素の状態（例：「オンショアの風がやや強め」「干潮に向かっており浅いエリアに注意」等）を、**あらかじめ用意した文章テンプレート**に当てはめて生成する。
- テンプレートの分岐条件（どの状態の時にどの文言を使うか）は`04_DATA_SPEC.md`または実装Repository側のロジック仕様に明記し、恣意的なブラックボックス処理にしない。
- 【必須】生成した要約テキストが誤解を招く安全上の断定表現（例：「今日は絶対安全」「確実に良い波が来る」）を含まないよう、テンプレート設計時に法務レビュー（Agent5_Legal）を通す。

### 4.5 データ欠損・異常値への対応【必須】

- 一部データソースが取得できない場合、該当要素のスコア計算から除外し、総合スコアは取得できた要素のみで再計算する（重みを取得できた要素間で再正規化する）。
- 取得できたデータ要素数が一定割合（初期値：50%未満）を下回った場合、総合スコアの表示を行わず「データ不足のため参考情報を表示できません」と明示する。
- 異常値（例：物理的にあり得ない波高・風速）を検知した場合は、その要素を除外し、`09_OPERATIONS.md`のログに記録する。

### 4.6 Phase2の完了条件（Definition of Done）

- 【必須】3スポットそれぞれについて、総合コンディションスコア（0〜100、5段階ラベル付き）が表示される
- 【必須】要約テキストがルールベースで生成され、UIに表示される
- 【必須】データ欠損時のフォールバック表示が実装されている
- 【必須】スコアリングロジックの重み・適正レンジが`04_DATA_SPEC.md`に文書化されている
- 【推奨】スポーツ種別ごとのスコア出し分けに対応している

## 5. Phase 3：Stripe・ロール仕様

### 5.1 Phase3の目的と原則【必須】

Phase3では、無料会員と有料会員（プレミアム）を区別する決済機能（Stripe）とロールベースのアクセス制御を導入する。

【必須】以下を原則とする。

- 決済インフラはStripeを使用する。Stripeは利用（課金発生）が無い限りコストが発生しない従量課金モデルであり、「運営コスト0円」の原則を損なわない。
- Stripeの秘密鍵（Secret Key）はクライアントコードに含めず、Cloudflare Pages Functions等のサーバーサイド環境変数として管理する（詳細は06_SECURITY_AND_PRIVACY.md）。
- 決済処理自体を自前実装せず、Stripe Checkout / Stripe Billing等のホスティッド機能を優先的に利用し、PCI DSS対応范囲を最小化する。

### 5.2 ロール定義【必須】

| ロール | 説明 | Phase |
|---|---|---|
| ゲスト（未ログイン） | ログインせずに基本情報（Phase1・Phase2の一部）を閲覧できる | Phase1〜 |
| 無料会員 | ログイン済み。お気に入り登録等の基本機能を利用できる | Phase1〜 |
| プレミアム会員 | 有料。詳細予測・広告非表示・優先通知等の付加機能を利用できる（付加機能の具体は`03_UI_GUIDELINES.md`で定義） | Phase3〜 |
| 情報提供者 | 認定制。現地情報を投稿できる（Phase4で詳細定義） | Phase4〜 |
| 管理者 | 運営者。コンテンツ管理・ユーザー管理・情報提供者の承認権限を持つ | Phase3〜（権限の実体はPhase4で拡張） |

【必須】ゲストでも閲覧可能な範囲（無料提供する情報の範囲）を明確にし、プレミアム会員限定機能との差分を`03_UI_GUIDELINES.md`に文書化する。これは法務ドラフト（Phase5）の特定商取引法表示・利用規約作成の前提情報となる。

### 5.3 認証方式【必須】

- 【必須】認証は無料枠で運用可能な認証サービス（例：Cloudflare Access、Firebase Authの無料枠、Supabase Authの無料枠等）から選定する。具体的な選定はAgent4_Auth.mdで詳細化する。
- 【必須】パスワードを自前で平文管理しない。認証サービス側のハッシュ管理・トークン管理に委譲する。
- 【推奨】ソーシャルログイン（Google等）に対応し、会員登録のハードルを下げる。

### 5.4 Stripe連携の基本フロー【必須】

1. ユーザーがプレミアム会員登録を選択
2. Stripe Checkoutへリダイレクト（クライアント側でStripe秘密鍵を扱わない）
3. 決済完了後、Stripe Webhookをサーバーサイド（Cloudflare Pages Functions等）で受信
4. Webhook受信をトリガーに、ユーザーのロールを「無料会員」→「プレミアム会員」に更新
5. サブスクリプション解約・支払い失敗時もWebhook経由でロールを自動的に「無料会員」に戻す

【必須】Webhookの署名検証（Stripe Signature検証）を必ず実装し、なりすましリクエストによる不正なロール昇格を防止する。

### 5.5 料金プラン（初期仕様）

| プラン | 価格 | 内容 |
|---|---|---|
| 無料会員 | ¥0 | 基本のスポット情報・簡易コンディションスコア閲覧 |
| プレミアム会員 | 【要決定】月額課金（具体的価格は運用開始前にDECISIONS.mdで確定） | 詳細予測・広告非表示・優先通知・情報提供者投稿への優先表示等 |

【必須】具体的な価格・プレミアム機能の最終確定前に、Phase5の法務ドラフト（特定商取引法に基づく表記、返金ポリシー）を先に整備し、整合性を取る。

### 5.6 Phase3の完了条件（Definition of Done）

- 【必須】無料会員／プレミアム会員のロール切り替えがStripe Webhook経由で自動的に行われる
- 【必須】Stripe秘密鍵・Webhook署名検証がサーバーサイドで安全に管理されている
- 【必須】ゲスト／無料会員／プレミアム会員でアクセス可能な機能範囲が明確に分離されている
- 【推奨】サブスクリプション解約・支払い失敗時のロール降格が正しく動作する

## 6. Phase 4：情報提供者制度仕様

### 6.1 Phase4の目的と原則【必須】

Phase4では、現地のショップ・インストラクター・常連利用者等を「情報提供者」として認定し、Phase1〜3で提供する外部データ・AIスコアだけでは分からない**現地のリアルタイム情報**（混雑状況、水質、離岸流・障害物、イベント情報等）を投稿できる制度を構築する。

【必須】以下を原則とする。

- 情報提供者制度は**任意参加・認定制**とし、無認定の一般ユーザーが直接投稿できる範囲とは明確に区別する（無認定ユーザーの投稿可否は`6.5`で定義）。
- 投稿内容は事実情報の共有を目的とし、他者への誹謗・広告目的の乱用投稿を防ぐモデレーション体制を設ける。
- 情報提供者の投稿はあくまで参考情報であり、AIスコアリング（Phase2）同様「保証ではない」ことをUI上で明示する。

### 6.2 情報提供者の認定フロー【必須】

1. ユーザーが「情報提供者になる」申請を行う（申請フォームで活動エリア・属性（ショップ／インストラクター／常連等）を入力）
2. 管理者ロールが申請内容を確認し、承認／却下を行う（Phase4初期では自動承認は行わず、必ず人による確認を挟む）
3. 承認された情報提供者には専用の投稿UIと「情報提供者」バッジが付与される
4. 【推奨】情報提供者の活動実績（投稿数、他ユーザーからの有用性評価等）を記録し、著しく低評価が続く場合は管理者が認定を取り消せる

### 6.3 投稿可能な情報の種類【必須】

| 情報種別 | 内容 | 表示期限（初期値） |
|---|---|---|
| 混雑状況 | 「現在やや混雑」等の現地感覚情報 | 投稿から3時間 |
| 水質・海況の注意情報 | 濁り、クラゲ、離岸流、危険物等 | 投稿から24時間（安全情報のため長め） |
| イベント・お知らせ | 大会、ローカルルール変更、ビーチクリーン等 | 投稿者が期限を指定可能 |
| その他コメント | 上記に当たらない現地情報 | 投稿から24時間 |

【必須】安全に関わる情報（水質・海況注意情報）は、他の情報種別よりも目立つUI表現（アイコン・色）で強調表示する。

### 6.4 モデレーション方針【必須】

- 【必須】不適切投稿（誹謗中傷、虚偽の安全情報、広告スパム等）を通報できる機能を全ユーザーに提供する。
- 【必須】通報が一定数（初期値：3件）を超えた投稿は自動的に非表示にし、管理者確認待ちとする。
- 【必須】虚偽の安全情報（例：実際には危険な状況を「安全」と偽って投稿する等）は、情報提供者資格の取り消し対象とする。この基準は法務ドラフト（Phase5）の利用規約に明記する。
- 【推奨】投稿内容の自動フィルタリング（NGワード検知等）は、有料AI APIを使わずルールベース（NGワードリスト・正規表現等）で実装する。

### 6.5 無認定ユーザーの投稿可否【必須】

- Phase4初期スコープでは、無認定の一般ユーザーによる自由投稿は実装しない。
- 【任意】将来的に「軽い現地レポート（例：混雑度を絵文字で選ぶだけ等の簡易フィードバック）」を無認定ユーザーにも開放することを検討可能とするが、Phase4初期スコープには含めない。

### 6.6 情報提供者への還元（インセンティブ）【任意】

- 【任意】情報提供者のショップ・サービスへのリンク掲載、プレミアム会員向け表示での優先掲載等、非金銭的インセンティブの導入を検討可能とする。
- 【必須】金銭的な報酬制度を導入する場合は、法務ドラフト（Phase5）および税務上の扱いを別途整理してから導入する。本仕様書の初期スコープでは金銭的報酬は含めない。

### 6.7 Phase4の完了条件（Definition of Done）

- 【必須】情報提供者の申請・承認フローが動作する
- 【必須】情報提供者が混雑状況・安全情報・イベント情報を投稿できる
- 【必須】通報機能とそれに基づく自動非表示ロジックが動作する
- 【必須】安全情報が他の情報種別より視覚的に強調される
- 【推奨】情報提供者の活動実績記録・認定取り消しフローが動作する

## 7. Phase 5：法務ドラフト

> 【重要な前提】本セクションは法務検討の**出発点となるドラフト**であり、弁護士等の専門家によるレビューを経ていない。実際の公開・運用前には、必ず専門家によるレビューを受けること。本ドキュメントおよび作成者は法的アドバイスを提供するものではない。

### 7.1 Phase5の目的と原則【必須】

Phase1〜4で構築した機能（無料データの要約・スコアリング表示、決済、情報提供者投稿）に対して、以下の法的整備が必要となる。

- 利用規約（Terms of Service）
- プライバシーポリシー（個人情報保護方針）
- 特定商取引法に基づく表記（Phase3のStripe決済に対応）
- 免責事項（気象・海況情報の「参考情報」性の明示）
- 情報提供者投稿に関する規約（Phase4対応）

### 7.2 免責事項の骨子【必須】

以下の要素を免責事項に必ず含める。

1. 本サービスが提供する波・風・潮汐・天候・コンディションスコア・要約テキストは、無料の公開データソースを基にした**参考情報**であり、その正確性・完全性・最新性を保証しない。
2. 本サービスの情報のみをもとにした海洋活動の安全判断は利用者自身の責任で行うものであり、運営者は生じた事故・損害について責任を負わない（ただし、運営者に故意または重大な過失がある場合を除く、等の一般的な限定表現は専門家確認のうえ調整する）。
3. 情報提供者による投稿情報についても、運営者が事実確認を行ったものではなく、参考情報として提供される。
4. データソースの提供元（気象庁、その他無料API提供元等）のクレジット表記・利用規約遵守状況を明記する（`04_DATA_SPEC.md`と整合させる）。

### 7.3 利用規約の骨子【必須】

利用規約には最低限、以下の項目を含める。

| 項目 | 内容の方向性 |
|---|---|
| サービス概要 | Phase1の1.2目的を引用・要約 |
| 会員種別 | Phase3の5.2ロール定義を引用 |
| 禁止事項 | 虚偽投稿、他者への誹謗中傷、システムへの不正アクセス、スクレイピング等の禁止 |
| 情報提供者規約 | Phase4の6章を引用し、投稿内容の権利帰属（運営者への利用許諾範囲）を明記 |
| 料金・解約 | Phase3の5.5料金プランを引用し、解約方法・返金ポリシーを明記 |
| 免責事項 | 7.2を引用 |
| サービス変更・終了 | 運営者が事前告知の上でサービス内容を変更・終了できる旨を明記 |
| 準拠法・管轄 | 日本法準拠、管轄裁判所の指定（専門家確認要） |

### 7.4 プライバシーポリシーの骨子【必須】

| 項目 | 内容の方向性 |
|---|---|
| 取得する個人情報 | メールアドレス、認証情報、決済情報（Stripe経由。カード情報自体は運営者側で保持しない）、お気に入り設定等の利用状況データ |
| 利用目的 | サービス提供、会員管理、決済処理、サービス改善のための利用状況分析 |
| 第三者提供 | Stripe（決済処理）、認証サービス提供元、無料データAPI提供元へのアクセス状況（個人情報の第三者提供に該当するか要確認） |
| Cookie等の利用 | PWAとしてのローカルキャッシュ、必要であればアクセス解析タグの利用有無を明記 |
| 開示・削除請求への対応 | 個人情報保護法に基づく開示・訂正・削除請求への対応窓口を明記 |
| 未成年者の利用 | マリンスポーツ利用者に未成年が含まれる可能性を踏まえ、保護者同意等の要否を専門家確認のうえ整理 |

### 7.5 特定商取引法に基づく表記の骨子【必須】

Stripeによるプレミアム会員課金（Phase3）に対応するため、以下を明記する（実際の記載事項は最新の特定商取引法要件に基づき専門家確認のうえ確定する）。

- 運営者名称・連絡先
- 料金（Phase3の5.5と整合）
- 支払方法・支払時期（Stripe決済）
- サービス提供時期
- 返品・キャンセルに関する特約（サブスクリプションの解約条件）

### 7.6 情報提供者投稿に関する規約【必須】

- 投稿内容の著作権は投稿者に帰属するが、サービス上での表示・キャッシュ・要約表示のために必要な範囲で運営者に利用許諾する旨を明記する。
- 虚偽の安全情報の投稿は、Phase4の6.4で定義した資格取り消し基準とあわせて、利用規約上の禁止事項として明記する。
- 情報提供者に金銭的報酬を導入する場合（Phase4の6.6、将来検討）は、業務委託契約・税務処理（源泉徴収等）を別途整理する必要がある旨を明記する。

### 7.7 Phase5の完了条件（Definition of Done）

- 【必須】免責事項・利用規約・プライバシーポリシー・特定商取引法表記のドラフトが作成されている
- 【必須】各ドラフトについて、専門家（弁護士等）レビューが完了するまで「ドラフト」であることが明示されている
- 【必須】Phase1〜4の各仕様（ロール、料金、情報提供者制度）と矛盾がない内容になっている
- 【推奨】未成年者利用、データ提供元のライセンス遵守等、専門家確認が必要な論点がリスト化されている
